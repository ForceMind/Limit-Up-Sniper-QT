# 涨停狙击手量化版需求文档

## 1. 产品定位

“涨停狙击手”继续作为项目名称，但产品核心从原先的涨停题材工具升级为可持续运行的 A 股量化分析系统。

系统目标不是简单展示新闻或推荐列表，而是形成完整闭环：

```text
自动抓取新闻 -> 自动同步行情 -> AI结构化分析 -> 多Agent评分博弈
-> 历史K线关联 -> 每日买入建议 -> 持仓与卖出建议
-> 模拟成交/交割 -> 复盘拟合 -> 参数优化
```

当前版本只做模拟交易和研究回放，不连接真实券商账户，不真实下单。后续如接入实盘，需要单独增加交易权限、风控、人工确认和合规审计。

## 2. 核心用户场景

### 2.1 盘前

- 自动汇总昨夜和盘前新闻。
- AI识别受影响行业、个股、利好利空程度和事件类型。
- 输出当天重点关注和计划买入列表。
- 后台显示数据覆盖情况：新闻是否最新、行情是否同步、AI是否分析完成。

### 2.2 盘中

- 自动拉取最新新闻和行情。
- 对新增新闻进行增量AI分析。
- 同步分时行情，判断是否触发买入、止盈、止损或风险卖出。
- 前台像交易软件一样显示账户、持仓、成交、交割单、推荐池。

### 2.3 盘后

- 固化当天新闻、行情、信号、模拟成交和账户快照。
- 生成复盘结果：哪些信号成功、哪些失败、收益和回撤。
- 自动拟合参数，给出是否应用新参数的建议。

### 2.4 管理维护

- 后台控制新闻抓取、行情同步、AI分析、模型拟合和任务调度。
- 查看日志、失败任务、成本、数据覆盖率。
- 一键部署、一键更新、一键重启服务。

## 3. 功能范围

### 3.1 自动化新闻抓取

必须支持：

- 财联社电报抓取。
- 东方财富等现有历史新闻数据读取。
- 统一新闻结构：`id/date/time/source/title/text/url/tags/raw_payload`。
- 去重：按来源、时间、标题、正文hash去重。
- 增量抓取：只拉取上次成功时间之后的数据。
- 抓取状态：最近成功时间、失败原因、抓取条数、入库条数。
- 手动触发：后台按钮和 API 均可触发。
- 自动清理：按配置保留最近 N 天原始新闻。

应该支持：

- 多来源插件化，后续可扩展 RSS、自定义接口、爬虫源。
- 新闻源健康度统计。
- 新闻按日期、来源、关键词、股票代码筛选。

验收标准：

- 系统启动后无需人工操作，可持续写入最新新闻。
- 前台能看到最新新闻列表，并明确显示“请求日期”和“实际数据日期”。
- 新闻源失败不会阻塞行情同步和交易回放。

### 3.2 行情和必盈数据同步

必须支持：

- 股票基础列表同步。
- 日K历史数据同步和补齐。
- 5分钟K线同步。
- 实时/近实时行情快照同步。
- 涨停池、跌停池、炸板池同步。
- 交易日历识别。
- 必盈接口限流和失败退避。
- 缓存状态展示：每个日期、每只股票是否有日K/分时K。

应该支持：

- 按事件股票同步。
- 按推荐股票同步。
- 按持仓股票同步。
- 按全市场股票分批同步。
- 缺失数据自动补齐队列。

验收标准：

- 盘中能自动同步当天推荐和持仓股票的分时K。
- 后台能看到必盈接口是否启用、额度/频控、最近错误。
- 回放时能区分真实分钟K成交和日线 fallback。

### 3.3 AI结构化分析

模型：

- 默认使用 `deepseek-v4-flash`。
- 模型、API key、费用参数必须可配置。
- API key 不应硬编码在代码或提交文档中，生产环境优先从环境变量读取。

每条新闻需要结构化输出：

- 情绪方向：利好、利空、中性。
- 情绪强度：`-1` 到 `1`。
- 事件类型：政策、业绩、订单、产品、技术、产业链、宏观、风险等。
- 影响行业。
- 影响个股。
- 个股影响方向：利好/利空/不确定。
- 个股影响强度：`0` 到 `100`。
- 影响周期：盘中、1日、3日、5日、20日。
- 置信度。
- 解释理由。

多Agent要求：

- 新闻情绪Agent：判断新闻情绪和强度。
- 事件影响Agent：判断事件类型、产业链扩散和影响范围。
- 个股映射Agent：把新闻映射到A股公司。
- 技术走势Agent：结合K线判断可交易性。
- 风控Agent：判断高开、过热、回撤、负面事件、ST/退市风险。
- 博弈合议Agent：综合多Agent结论，输出最终买入/卖出评分。

AI分析要求：

- 支持缓存，避免同一新闻重复收费。
- 支持失败重试和降级规则分析。
- 支持AI费用统计和每日阈值告警。
- 分析结果可在后台查看原文、Prompt、模型、Token、耗时、结构化结果。

验收标准：

- 新新闻入库后自动进入AI分析队列。
- AI失败时仍能用规则分类生成低置信事件。
- 前台推荐必须能追溯到对应新闻和AI理由。

### 3.4 量化评分和推荐

系统需要输出每只股票短期上涨/下跌评分。

买入分核心因子：

- 新闻情绪分。
- 事件影响分。
- 技术走势分。
- 历史关联分。
- 风险可交易分。

卖出分核心因子：

- 负面新闻。
- 止盈止损。
- 技术转弱。
- 持仓时间。
- 事件兑现失败。
- 风险事件。

参数必须可调：

- Agent权重。
- 买入阈值。
- 观察阈值。
- 卖出阈值。
- 止盈止损。
- 最大持仓数。
- 单票资金。
- 最大持仓天数。
- 分时触发规则。
- 滑点和手续费。

验收标准：

- 前台显示每日买入列表。
- 每个推荐都能看到新闻、事件、Agent拆分和理由。
- 后台可修改参数并即时影响回放和推荐。

### 3.5 全周期回放和分时回放

必须支持：

- 严格时间穿越控制：某天只能使用该日期及以前的数据。
- 信号日和成交日分离。
- 日线回放：次日开盘买入，收盘判断卖出。
- 分时回放：新闻时间后下一根5分钟K模拟买入。
- 盘中止盈止损。
- 涨跌停不可成交或按保守规则处理。
- A股T+1。
- 手续费、印花税、过户费。

验收标准：

- 可以从任意日期开始线性回放到指定日期。
- 能查看每一笔买入、卖出、未成交原因。
- 能看到收益曲线、最大回撤、胜率、平均收益、交易次数。

### 3.6 模拟账户、持仓、成交和交割单

前台必须像真实交易软件：

- 账户总资产。
- 可用资金。
- 持仓市值。
- 当前持仓。
- 可用数量。
- 持仓成本。
- 最新价。
- 浮动盈亏。
- 今日成交。
- 历史成交。
- 交割单。
- 已实现盈亏。
- 每日清算汇总。

交割单字段：

- 交割日期。
- 成交时间。
- 证券代码。
- 证券名称。
- 买卖方向。
- 成交数量。
- 成交价格。
- 成交金额。
- 佣金。
- 印花税。
- 过户费。
- 清算金额。
- 已实现盈亏。

验收标准：

- 当前持仓可以由历史成交重建。
- 切换历史日期时不能显示未来持仓。
- 每笔成交和交割单一致。

### 3.7 前台页面

前台定位：交易终端。

必须包含：

- 顶部账户摘要。
- 最新新闻列表。
- 结构化事件列表。
- 今日买入计划。
- 当前持仓。
- 历史成交。
- 交割单。
- 历史计划收益。
- 数据日期提示。

交互要求：

- 默认选择系统当天。
- 如果当天没有数据，明确提示实际数据日期。
- 表格适合大量扫描，不做营销页。
- 推荐、新闻、成交之间可追溯。

### 3.8 后台管理

后台定位：系统控制台。

必须包含：

- 数据源配置：新闻源、必盈配置、AI配置。
- 任务调度：自动抓新闻、同步行情、AI分析、回放、拟合。
- 策略参数配置。
- 模型拟合和候选参数比较。
- 数据覆盖率。
- AI调用成本。
- 运行日志。
- 错误队列。
- 手动触发按钮。
- 备份和恢复。
- 部署/更新状态。

应该包含：

- 用户和权限模块可以保留接口边界，但量化核心先不依赖商业会员体系。
- 邮件/消息通知：数据失败、服务异常、当日推荐生成完成。

### 3.9 自动调度

沿用旧项目“智能时间表”的思路：

```text
08:30-09:30 盘前准备：补新闻、补日K、生成盘前计划
09:30-11:30 盘中：高频新闻抓取、行情同步、增量分析
13:00-15:00 盘中：继续增量处理和持仓风控
15:00-15:30 收盘固化：同步收盘行情、更新模拟账户
15:30-18:00 盘后复盘：回放、拟合、生成复盘
夜间       低频检查、备份、历史数据补齐
```

任务要求：

- 每个任务有状态、最近开始、最近结束、耗时、失败原因。
- 同一任务不能并发重入。
- 失败自动退避。
- 后台可以暂停、恢复、立即执行。

### 3.10 一键部署和更新

必须提供：

- `scripts/install_server.sh`：服务器首次安装。
- `scripts/update_server.sh`：拉取/覆盖代码、安装依赖、迁移配置、重启服务。
- `scripts/restart_server.sh`：重启服务。
- `scripts/backup_data.sh`：备份 `backend/data`。
- `scripts/restore_data.sh`：恢复数据。
- `systemd/qt.service`：生产守护进程模板。
- `.env.example`：环境变量模板。

部署要求：

- 不覆盖生产数据。
- 不把API key写入仓库。
- 支持测试端口启动。
- 支持健康检查。
- 更新失败可以回滚到上一次备份。

## 6. 非功能要求

- 稳定性：服务可连续运行，不因单个数据源失败崩溃。
- 可观测性：所有自动任务有日志、状态、错误原因。
- 可恢复性：数据目录可备份恢复。
- 可解释性：每个推荐能追溯新闻、事件和Agent评分。
- 可控性：所有关键参数后台可调整。
- 性能：前台常用接口在本地数据下尽量 2 秒内返回；重型回放后台异步化。
- 安全：密钥走环境变量或本地私有配置，部署脚本不打印敏感信息。
