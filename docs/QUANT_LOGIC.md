# 量化系统当前买卖逻辑

本文描述的是当前已经实现的逻辑。系统已经支持基于本地5分钟K线的分时回放，但还没有接入盘口快照、逐笔成交和真实委托撮合。

## 1. 数据输入

系统当前只使用本地历史数据：

- `backend/data/news_history.json`：历史新闻。
- `backend/data/news_analysis_records.json`：历史AI分析记录。
- `backend/data/biying_stock_list.json`：股票代码和名称映射。
- `backend/data/quant_data.sqlite3` 的 `market_daily_bars`：每只股票的日 K 主数据。
- `backend/data/kline_day_cache/*.json`：旧日 K JSON 缓存，只作为兼容读取和迁移来源。

当前没有实时下单，也没有真实账户接口。

前台交易终端使用模拟券商账户视图：

- `GET /api/quant/trading_account`：返回账户资产、持仓、历史成交、交割单和每日清算汇总。
- 手续费默认按佣金 `0.025%`、最低 `5` 元、过户费 `0.001%`、卖出印花税 `0.05%` 估算。
- 持仓成本包含买入佣金和过户费；卖出已实现盈亏会扣除卖出佣金、过户费和印花税。
- 当前仍是纸面撮合，不连接真实券商账户，不会真实下单。

## 系统运行策略与策略库模型

系统里有两类策略视图，不能混为一个账户：

- 系统运行策略：后台任务当前使用的全局参数，来源于运行配置和“策略参数”。它驱动系统启动、每日推荐、模拟交易任务和后台“持仓成交”的系统运行账户。它是当前线上运行态，可以被后台配置修改。
- 策略库模型：进化/回测后保存下来的参数组合，每个模型应该有自己的收益、回撤、胜率、成交、交割单和训练痕迹。它是历史实验结果，不会自动改变系统运行账户，除非后续显式“应用为系统运行策略”。
- 资金档策略：1万、2万-5万、5万-10万等是按资金规模派生出的约束版本，主要控制最大持仓、单票资金和是否能买一手。它们不是独立训练出来的模型，但可以作为前台用户默认跟随策略。

因此，后台“持仓成交”看的是系统运行账户；“策略库”里点“查看交割单”看的是某个策略模型自己的回测账户；前台用户登录后看到的是“用户资金 + 用户跟随策略”重放后的账户。

系统运行策略参数保存在 `backend/data/quant_state.json` 的 `strategy_params` 和 `strategy_source`。来源可能是默认参数、后台手动保存、参数拟合、策略库模型应用、策略进化自动应用最佳模型，或资金档策略应用。策略库里的模型只有被明确“应用为系统运行策略”后，才会影响后台模拟交易任务。

## 2. 新闻事件结构化

每条新闻或AI分析记录会被转成 `NewsEvent`：

- `event_type`：政策催化、业绩财报、订单合作、产品技术、板块异动、宏观市场、风险事件等。
- `industry`：AI算力、半导体、电力能源、新能源、汽车、机器人、消费零售等。
- `code/name`：影响个股。
- `sentiment`：情绪分数，范围约为 `-1` 到 `1`。
- `impact_score`：事件影响分，范围 `0` 到 `100`。
- `ai_score`：历史AI分析里已有的个股分数。

原始新闻用股票名称匹配影响个股；AI分析记录优先用其中已经输出的股票代码。

## 3. Agent评分

每只股票在某个日期会由四个内部Agent打分：

- 新闻情绪Agent：看正负面关键词、历史AI分、新闻情绪强度。
- 事件影响Agent：看事件类型、行业、事件影响分，以及同类历史事件兑现表现。
- 技术走势Agent：看当时日K技术状态，包括3日/5日/20日收益、量比、波动率、回撤。
- 风控Agent：看波动率、回撤、过热程度、负面事件。

最终买入分：

```text
buy_score =
  新闻情绪分 * sentiment_weight
  + 事件影响分 * event_weight
  + 技术走势分 * technical_weight
  + 风控可交易分 * risk_weight
```

初始权重：

```text
sentiment = 0.35
event = 0.25
technical = 0.25
risk = 0.15
```

系统运行模拟交易后，会根据高分区间的历史表现微调权重。

当前参数已经可以在页面和接口中调整：

- Agent 权重：新闻情绪、事件影响、技术走势、风控。
- 买卖阈值：买入阈值、观察阈值、卖出评分阈值、回避阈值。
- 交易参数：止盈、止损、最大持仓天数、最大持仓数量、每日候选数量、单票资金上限。
- 评分系数：情绪系数、AI分系数、事件影响/历史统计占比、历史收益/胜率系数、负面情绪和技术风险卖出系数。

参数接口：

- `GET /api/quant/strategy_params`
- `POST /api/quant/strategy_params`
- `POST /api/quant/strategy_params/reset`
- `POST /api/quant/fit_strategy`

拟合逻辑会用当前线性回放，对“当前参数、进攻型、保守型、短线快进快出、事件驱动、风控优先”等候选参数进行比较。目标函数目前是：收益 - 回撤惩罚 + 胜率奖励 + 成交样本奖励。拟合结果可以直接写回策略参数。

## 4. 买入规则

当前有两套买入模拟：

### A. 静态推荐

接口：

```text
GET /api/quant/recommendations
GET /api/quant/daily_plan
```

规则：

- `buy_score >= 72`：买入候选。
- `60 <= buy_score < 72`：重点观察。
- `buy_score < 60`：暂不买入。
- 如果 `sell_score >= 70` 且 `buy_score < 65`：回避/卖出。

### B. 全周期线性回放

接口：

```text
GET /api/quant/timeline
```

这是更接近真实使用的逻辑：

1. 按交易日从早到晚推进。
2. 当天只能看到当天及以前的新闻、已兑现的历史样本、当时已有的日K。
3. 当天新闻产生买入信号。
4. 信号不会当天成交。
5. 下一交易日按开盘价模拟买入。
6. 每天收盘后更新持仓、收益和卖出判断。

示例：

```text
5月1日：新闻出现，系统产生买入信号。
5月2日：用开盘价模拟买入。
5月2日收盘：计算当日持仓收益，判断是否卖出。
5月3日及以后：继续按规则更新，直到卖出。
```

## 5. 卖出规则

当前全周期回放和模拟持仓的卖出条件：

- 止损：持仓收益 `<= -5%`。
- 止盈：持仓收益 `>= 8%`。
- 持仓到期：默认持有 `3` 个交易日。
- 卖出评分触发：当日重新评分后 `sell_score >= 72`。

卖出价格：

- 当前使用卖出日的收盘价模拟。

## 6. 分时逻辑现状

当前已经接入本地5分钟K缓存：

- 目录：`backend/data/kline_cache/*.csv`
- 格式：`time,open,close,high,low,volume,amount`
- 接口：`GET /api/quant/intraday_timeline`
- 必盈同步：`POST /api/data/biying/sync_intraday?date=YYYY-MM-DD&source=events&max_codes=200`

分时回放规则：

1. 新闻信号有明确时间戳。
2. 如果该股票当天有5分钟K，买入价使用“信号时间之后下一根5分钟K的开盘价”。
3. 持仓期间逐根检查5分钟K：
   - `low <= entry_price * 0.95`：按止损价卖出。
   - `high >= entry_price * 1.08`：按止盈价卖出。
   - 如果同一根5分钟K同时触发止盈和止损，保守假设止损先发生。
4. 如果当天没有分钟K，但开启 `use_daily_fallback=true`，则使用日线开盘/收盘价兜底。
5. 结果中会区分：
   - `intraday_trade_count`：真实使用5分钟K成交/卖出的数量。
   - `fallback_trade_count`：使用日线fallback的数量。

注意：当前本地分钟K缓存只有少数日期，比如 `2026-03-10` 和 `2026-05-19`。服务器试运行后，可以先用必盈同步接口按新闻事件股票补齐当日分钟K；要覆盖更长历史，则需要持续抓取并保存更多交易日的分钟K。

## 7. 下一步分时版本应该怎么做

要做更接近真实交易的分时模拟，需要继续接入盘口快照：

- 09:25 到 09:30：集合竞价信号和开盘价格处理。
- 09:30 后：按分钟K或实时价触发买点。
- 买入价格：信号后的下一根分钟K开盘价，或实时快照价。
- 止盈止损：盘中实时触发，而不是等收盘。
- 卖出价格：触发分钟的下一根分钟K开盘价，或实时快照价。
- 交易约束：涨跌停不可成交、T+1、仓位限制、滑点、手续费。

当前代码已经把“信号、成交、持仓、卖出”分开了，后续可以把日K价格替换成分钟K价格，不需要推翻主结构。
