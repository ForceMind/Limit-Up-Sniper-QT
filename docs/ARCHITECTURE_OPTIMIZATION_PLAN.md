# 架构优化与持续进化计划

本文是当前项目从“能跑”进入“长期可复用量化系统”的执行计划。目标不是把所有逻辑堆到页面按钮里，而是形成稳定的数据层、策略层、回测层、进化层和展示层。

### 0.1 不合理的功能设计

- `backend/app/main.py` 不应该继续新增业务逻辑。入口文件只应创建 FastAPI、注册中间件、注册路由和托管静态文件；认证、前台、后台、任务、数据、量化接口要拆到独立 router。
- `backend/app/quant/engine.py` 不应该同时做数据仓库、因子计算、策略评分、回放和账户。长期看会导致一个修复影响全系统，也很难定位服务器性能问题。
- 前后台页面不应该继续堆单文件 Vue。后台用户管理已经需要独立弹窗或页面，后续数据管理、策略模型、任务日志也应该有单独模块。
- 自动任务不应该只靠进程内线程表达状态。系统要长期运行进化模型，任务状态、进度、最后错误和下一次运行时间必须落库。
- 数据层不应该继续“SQLite + 散落 JSON 双主”。JSON 只能作为迁移来源和兼容缓存，长期主存储应收敛到 SQLite，未来数据量扩大后再迁 PostgreSQL。
- 部署更新不应该每次重复做全量迁移或反复改 Nginx。日常 `qt update` 应默认做智能验证，只有首次、数据库缺失、迁移脚本变化或显式 `qt migrate` 才全量合并历史文件。
- 前台接口不应该登录后一次性拿全量数据。首页只拿概览、当前账户、当前策略和少量新闻；成交、交割单、回测明细、模型记录按需分页加载。
- “样例数据”过滤不应该散落在页面和多个接口里。样例识别应集中在数据层或仓库层，业务接口默认不返回样例，后台保留清理入口和诊断入口。

## 1. 当前问题判断

### 1.1 16 个策略收益没有差异的原因

已确认的代码级原因：

- 遗传进化会变异 `sentiment_weight`、`event_weight`、`technical_weight`、`risk_weight`。
- 但实际 Agent 打分里使用的是全局 `model_weights()`，没有读取候选策略自己的临时参数。
- 因此候选策略的权重基因没有真正参与回测，只剩阈值、止盈、止损、持仓天数等少量参数在变化。
- 如果服务器数据覆盖不足、闭环交易数过少，多个候选很容易全部收益为 `0` 或收益接近。

已修正方向：

- Agent 打分改为使用当前策略参数中的四类权重。
- 遗传进化默认使用分时回放，保留完整成交、交割单和资金曲线。
- 引入第一批扩展因子，让候选策略不仅调阈值，也能调因子偏好。

### 1.2 系统没有持续进化的原因

当前调度器已经周期运行新闻、AI、日K补齐、龙虎榜、模拟交易和策略复盘，但策略进化只在后台点击按钮时运行，没有纳入常驻调度。

已修正方向：

- 新增 `strategy_evolution` 调度任务。
- 默认不自动开启 `STRATEGY_EVOLUTION_ENABLED`，避免小服务器刚启动就运行重型回测导致内存占满；确认数据覆盖和内存稳定后再显式开启。
- 默认每 6 小时运行一次，可用环境变量调整。
- 进化运行中或暂停时不会重复启动。

## 4. 执行原则

- 数据补齐优先于复杂模型，否则收益全 0 无法判断策略优劣。
- 分时回放优先，日K只做兜底。
- 每个模型必须可复用：参数、收益、回撤、胜率、交易、交割单都要保存。
- 自动进化默认不自动应用最佳模型，避免服务器无人值守时把线上策略突然切换。
- 所有长任务必须有进度和中文日志，不走同步大请求。

## 6. 训练轨迹必须可审计

遗传进化不能只保存最后 16 个模型，否则看不到“跑过多少、怎么淘汰”。新的记录链路如下：

```text
strategy_runs
  一次训练任务：时间、区间、代数、种群、最佳结果

strategy_model_metrics
  每一代摘要：最佳目标、收益、回撤、胜率、评估数量、保留数量、淘汰数量、精英线

strategy_candidates
  每个候选策略：run_id、generation、rank、selected、elimination_reason、objective、收益、回撤、胜率、闭环交易、参数指纹、完整参数

strategy_models / strategy_model_records
  最终可复用模型，以及该模型对应的成交、交割单、每日清算
```

例如跑 100 代、每代 20 个候选，`strategy_candidates` 应产生约 2000 条候选记录。后台“策略库 -> 训练过程与淘汰记录”可以查看最近训练批次、每代保留线、保留/淘汰数量、候选排名和淘汰原因。

淘汰原因当前按可解释规则归类：闭环交易不足、收益为负、最大回撤过大、盈亏比不足、夏普为负、综合目标函数排名低于精英线。后续如果加入更多因子，淘汰原因也要跟着细化，例如“流动性不足”“龙虎榜席位负贡献”“题材强度不足”。

## 7. 更大数据量与迁移路线

当前阶段 SQLite 是主库，JSON 只保留为兼容导入源和轻量状态文件。更大规模时不要直接把 JSON 越堆越大，按下面路线升级：

| 阶段 | 适用规模 | 存储方案 | 说明 |
|---|---:|---|---|
| 当前 | 单机、几十万到几百万行 | SQLite WAL | 足够支撑新闻、日K、分钟K、龙虎榜、训练轨迹和后台查询 |
| 扩展一 | 多进程任务、多人访问 | PostgreSQL | 用户、任务、策略、训练轨迹迁入 PG，保留事务和权限控制 |
| 扩展二 | 大量分钟K/盘口/逐笔 | ClickHouse 或 Parquet 分区 | 行情明细按日期和股票分区，回测按日期范围顺序扫描 |
| 扩展三 | 多机器训练 | 任务队列 + Worker | Web 只接收请求和展示状态，训练/补数据由独立 worker 执行 |

迁移规则：

- 所有可复用数据必须有主键或自然唯一键，上传数据包只做 upsert 合并，不覆盖服务器已有数据。
- 备份包包含 SQLite、配置脱敏快照、导入日志和版本号；密钥不进入 Git，也不进入普通导出包。
- 训练轨迹可以按 run_id 分批导出，未来迁移到 PostgreSQL 时保持表名和字段语义一致。
- 分钟K和更细行情后续可以从 SQLite 拆出到列式存储，但策略结果、用户和任务状态仍保留在事务型数据库。

## 8. 因子融合路线

已经进入模型的因子：新闻情绪、事件类型与影响分、历史兑现、日K动量、放量、突破、龙虎榜净买入和席位活跃度。

建议逐步加入，但每个因子都必须满足“有历史数据、不会未来函数、能做消融测试”：

- 市场环境：指数趋势、上涨家数、涨停/跌停数量、成交额、赚钱效应。
- 板块题材：概念强度、同题材涨停数量、龙头高度、板块扩散速度。
- 流动性：成交额、换手率、量比、近 N 日成交稳定性、小资金可买入约束。
- 涨停结构：首板、二板、断板反包、炸板回封、开板次数。
- 竞价与分时：竞价涨幅、竞价金额、开盘承接、分时放量、冲高回落。
- 龙虎榜席位：席位近期胜率、隔日溢价、席位协同、游资/机构风格标签。
- 新闻语义：事件新鲜度、重复报道强度、来源权重、行业链条、风险语义。
- 风控过滤：ST/退市风险、监管问询、减持、解禁、财务异常、流动性不足。

落地顺序建议是：先补数据覆盖，再加入因子表，再把因子快照写入回测记录，最后让进化基因控制因子权重。这样前台看到的每个策略，不只是收益不同，也能解释“它偏好什么因子”。
